# Critical / High 脆弱性の具体例 — ラベル別ダイジェスト

*`data/ethereum_vulns.csv`(全2,225件)のうち `severity` が `Critical`(3件)/
`High`(63件)の66件を対象に、`label`(プロトコル領域)ごとに具体的な内容を
まとめたもの。各項目は原則として以下の形式で書いている。*

> 〇〇なことが原因で、攻撃者がXXというメッセージ/トランザクションを送るだけで
> OOできる脆弱性。

**読む前に**: `root_cause` / `attack_path` / `label` はパスルール+LLM
(`gemma3:31b`)による自動分類で、人手検証はしていない
([`limitations.md`](./limitations.md) §3)。また高severityの一部は
リリースノートの定型文(例:「今回は重大なセキュリティ修正を含みます」)しか
情報が無く、具体的な攻撃手順を再構成できない。そうしたエントリは
本文中で「詳細非公開」と明記するか、末尾の「ノイズ・除外エントリ」節にまとめ、
存在しない詳細を捏造しないようにしている。

---

## Critical(3件、全件)

- **[Besu] `CALL`/`DELEGATECALL`のガス計算での32bit型変換ミスが原因で**、
  攻撃者が残余ガス量によって成否が分かれるようなコントラクト呼び出しを
  含むトランザクションを送るだけで、実装ごとに異なる`stateRoot`を
  算出させ**コンセンサス分裂**を引き起こせる脆弱性
  (`GHSA-4456-w38r-m53x`, CVE-2022-36025)。単一EVM実装のネットワークでは
  想定超過のガスで実行させることも可能。goevmlab(差分ファジング)による
  監査で発見、実運用networkでの悪用は未確認。

- **[Geth] 依存するGoランタイム自体の脆弱性(CVE-2020-28362)が原因で**、
  Go `<1.15.5`/`<1.14.12`でビルドされたノードが、特定の入力を処理する
  だけでクラッシュしうる脆弱性(`GHSA-m6gx-rhvj-fh52`)。Geth自体のコードには
  問題がなく、Goのバージョンを上げてリビルドするだけで解消する
  — 依存チェーン由来のリスクの典型例。

- **[Teku] 同梱するlog4jライブラリのJNDIルックアップ機能が原因で**、
  攻撃者が`${jndi:ldap://...}`のような文字列をログに記録させるだけで
  リモートコード実行(RCE)が成立し、バリデータの署名鍵にアクセスされ
  うる脆弱性(Log4Shell, `GHSA-mwfw-vm54-g3p7`, CVE-2021-44228)。Tekuの
  ログ利用方法により実際の悪用可能性は限定的と判断されたが、念のため
  緊急パッチ(21.12.1)がリリースされた。

## p2p — devp2p / `eth`ワイヤプロトコル(execution層、9件)

- **[Geth] `GetBlockHeadersRequest`の`count`パラメータに下限チェックが
  無かったことが原因で**、攻撃者が`count=0`を指定した1個のメッセージを
  送るだけで`count-1`が整数アンダーフローし、ノードに大量メモリを
  確保させてクラッシュ/DoSさせられる脆弱性(CWE-190)。

- **[Geth] pingリクエスト受信ごとに新しいgoroutineを無制限に生成して
  いたことが原因で**、攻撃者が大量のping要求を送りつけるだけで
  goroutineが際限なく増殖し、メモリ枯渇(OOM)でノードを
  クラッシュさせられる脆弱性(修正: v1.12.1)。

- **[Reth] libp2pの新規ストリームオープンにリソース上限が設けられて
  いなかったことが原因で**、攻撃者ノードが接続を継続的に開き続ける
  だけで小さなメモリ確保が積み重なり、被害ノードをOOM Killさせられる
  脆弱性(CWE-770)。同種の実装不備はlibp2pベースの他クライアント
  (後述のgrandine/lighthouse)にも共通して見つかっている。

- **[Geth] WebSocketフレーム長の扱い(依存ライブラリ`gorilla/websocket`)
  に整数オーバーフローがあったことが原因で**、攻撃者が細工した長さ
  フィールドを持つWebSocketフレームを送るだけでDoSを引き起こせる
  脆弱性(CWE-190)。

- 「高CPU負荷を発生させるp2pメッセージ」「クラッシュを起こすp2p
  メッセージ」の2件(CWE-400 / CWE-248)は、Ethereum Foundation Bug
  Bounty経由の報告でパッチのみ先行公開され、**技術的詳細は非公開**
  (`"More details to be released later"`)。存在は確認できるが再現手順は
  再構成できない。

## rpc — JSON-RPC / GraphQL サーフェス(7件)

- **[Geth] GraphQLエンドポイントにクエリの複雑度・コスト制限が実装
  されていなかったことが原因で**、`--http --graphql`が有効なノードに
  攻撃者が巧妙に細工したGraphQLクエリを送るだけで、メモリを
  食い潰しデーモンをハングさせられる脆弱性(CWE-400)。ベンダー側は
  「このGraphQLエンドポイントは悪意あるクライアントへの耐性を
  想定して設計していない」とコメントしている。

- **[Erigon] 依存するJSONパーサー(`jsonparser`)の境界チェック不備が
  原因で**、攻撃者が細工したJSON入力を渡すだけで範囲外読み取り
  (out-of-bounds read)を引き起こせる脆弱性(CVE-2026-32285, CVSS 7.5,
  CWE-125)。

- **[Geth] `debug_traceChain`(旧`TraceChain`)が終了ブロック番号が
  開始ブロック番号以降であることを検証していなかったことが原因で**、
  攻撃者(または利用者)が終了<開始となるようなRPCリクエストを送る
  だけで想定外の負荷/異常動作を引き起こしうる脆弱性(CWE-20,
  geth < 1.8.14)。

- **[Reth]** ライブ同期中の特定の状態遷移でpanicするか不正な状態
  (bad state)に陥る重大バグが報告されているが、**再現条件の詳細は
  非公開**(v0.1.0-alpha.21のリリースノートでの言及のみ)。

## crypto — ハッシュ/署名/secp256k1・BLS実装(6件)

- **[Geth] secp256k1曲線上の点`(x, y)`について、`x`・`y`がともに有限体の
  位数`P`未満であることを検証していなかったことが原因で**、攻撃者が
  範囲外の座標を含む公開鍵/署名データを1個のp2pメッセージ(RLPx
  ハンドシェイクないし署名検証パス)として送るだけで、フィールド要素の
  不変条件が破れ未定義動作(パニック/クラッシュ)を引き起こし、ノード
  プロセス全体を落とせる脆弱性(`GHSA-2gjw-fg97-vg3r`)。修正はGo実装
  (`curve.go`, `signature_nocgo.go`)とCラッパー(`ext.h`)の両方で、
  境界チェック追加と戻り値の未チェック修正が同時に行われている。
  Geth v1.16.9 / v1.17.0で修正。

- **残り5件**は`golang.org/x/crypto/ssh`パッケージ(Go標準の拡張ライブラリ)
  のCVE群(不正なホストキー検証によるMITM、GSSAPI経路でのnilポインタ
  参照、空平文パケットでのpanicなど)。**Gethは自前でSSHサーバー機能を
  提供していない**ため、これらは依存関係スキャン(govulncheck)経由で
  拾われた「依存グラフ上に存在するCVE」であり、実際のEthereumノードの
  攻撃対象領域として到達可能かは本データセットからは確認できない
  ([`limitations.md`](./limitations.md) の「Dependency-bump rows」の
  注記を参照)。

## p2p-interface — gossipsub / req-resp(consensus層、6件)

- **[Reth/Grandine/Lighthouse] libp2pの新規ストリームオープンに対する
  リソース制限が設けられていなかったことが原因で**、攻撃者ノードが
  ストリームを大量に開き続けるだけで小さなメモリ確保が積み重なり、
  被害ノードをOOM Killさせられる脆弱性(CWE-770、上記p2pの`reth`の件と
  同一クラスの不備がlibp2pベースの複数クライアントで独立に修正)。

- **[Teku] 依存するNettyライブラリのHTTP/2リソース管理不備
  (CVE-2021-37136 / CVE-2021-37137)が原因で**、攻撃者が細工した
  p2pメッセージ/HTTP2フレームを送るだけでリソース枯渇によるDoSを
  引き起こせる脆弱性。

- **[Lighthouse] BLS署名検証の中核となる暗号ライブラリ`blst`に既知の
  脆弱性が存在していたことが原因で**、`v1.2.0`未満のまま起動して
  いるノードは攻撃者から細工されたデータを送られるだけで影響を
  受けうる状態にあった(2021年4月の"Finalized #25"インシデントに
  関連する`blst`セキュリティアドバイザリ)。※このデータセット上の
  リリースノートには具体的な再現手順までは書かれていない。

## EVM / opcodes — インタープリタ・命令セマンティクス(evm 3件 + opcodes 3件)

- **[Geth] EVMインタープリタの`RETURNDATA`(呼び出し結果データ)を扱う
  メモリ管理にメモリ破損バグがあったことが原因で**、攻撃者が特定の順序
  で呼び出し・メモリ操作を組み合わせたコントラクトを呼ぶトランザクション
  を送るだけで、ノードごとに異なる`stateRoot`を算出させられる脆弱性
  (`GHSA-9856-9gg9-qcmq`)。**2021年8月22日、ブロック13107518で実際に
  メインネット上で悪用され、マイノリティチェーンフォークが発生した**
  (Geth v1.10.8で修正、ポストモーテム公開済み)。

- **[Geth] `MULMOD`命令が内部で使う`uint256`ライブラリが、モジュロに
  `0`を渡すケースの境界チェックを欠いていたことが原因で**、攻撃者が
  `mulmod(a, b, 0)`を実行するだけのコントラクトを呼ぶトランザクション
  を送るだけで、内部バッファへの範囲外アクセス(`index -1`)により
  panicを起こし、ノードをネットワークから脱落させられる脆弱性
  (`GHSA-jm5c-rv3w-w83m`)。

- **[Besu] `SHL`/`SHR`/`SAR`命令の実装で32bit符号付き整数への型変換を
  誤ったことが原因で**、攻撃者が約20億〜40億という(意味は無いが値として
  は妥当な)巨大シフト量を指定するトランザクションを送るだけで、実行が
  異常終了し検証に失敗、複数クライアントが混在するネットワークでは
  **フォーク**を引き起こせる脆弱性(CVE-2021-41272)。※アドバイザリと
  CVEレコードの2件として重複記録されている。

- **[Geth] `cmd/evm`(スタンドアロンのEVM実行/デバッグツール)がバイト
  コードの境界チェックを行っていなかったことが原因で**、攻撃者が
  細工したバイトコードを渡すだけでSEGVを引き起こしプロセスをクラッシュ
  させられる脆弱性(CWE-119)。※本番ノードのライブ実行パスではなく
  開発用ツールが対象。

- **[Geth] EVMメモリ上の動的配列長チェックが不十分だったことが原因で**
  (CVE-2018-20421)、攻撃者が`assembly { mstore }`で配列長を書き換えた
  上で巨大なインデックスに書き込むトランザクションを送るだけで、
  大量メモリ確保によりノードをDoSできる脆弱性。

## transactions — トランザクション種別・検証(2件)

- **[Geth] `selfdestruct`直後に生成される内部オブジェクトの`balance`
  引き継ぎ処理(`createObject`)を、あるPRで意図せず変更していたことが
  原因で**、攻撃者が「同一アドレスへ自爆(self-destruct)させた直後に、
  同一トランザクション内でさらに送金する」という特定順序のトランザクション
  を送るだけで、Geth独自の誤った残高を算出させ**コンセンサス分裂**を
  引き起こせる脆弱性(`GHSA-xw37-57qp-9mm4`, 2020年8月報告, v1.9.20で修正)。
  最小修正は`prev.deleted`のチェックを1行追加するだけだった。

- **[Nethermind/Juno] Starknetの`cairo-lang-starknet-classes`ライブラリに
  あるSierraバイトコード展開処理に整数オーバーフローがあったことが
  原因で**、攻撃者が細工した`Declare v2/v3`トランザクションを送るだけで
  無限ループを発生させ、高CPU負荷でノードをDoSできる脆弱性
  (CVE-2025-29072)。

## sync — スナップ同期/LES(2件)

- **[Geth] LESプロトコルの`GetBlockHeadersMsg`ハンドラで`Skip`値の
  符号なし変換に誤りがあったことが原因で**、攻撃者が`query.Skip = -1`
  を指定したパケットを1個送るだけで配列アクセス違反を引き起こし、
  ノードを即座にクラッシュさせられる脆弱性(geth < 1.8.11)。通称
  **"Ethereum Packet of Death"(EPoD)**として知られる。

---

## beacon-chain:sync-committee — Electraのepoch処理(1件)

- **[Lighthouse] Electraフォークのepoch処理(`process_epoch`)における
  実効残高(effective balance)計算に誤りがあったことが原因で**、
  Electra対応ネットワークが通常のエポック遷移を進めるだけで、一部の
  Lighthouseノード(`v7.0.0-beta.0`〜`beta.4`)だけが正規チェーンと異なる
  実効残高を算出し、正規チェーンから孤立(フォーク)しうる脆弱性
  (`GHSA-wm9c-xvqq-5c28`)。Lighthouseはバリデータの約1/3を占めるため、
  もし悪用されればネットワークのファイナリティが停止しかねなかった。
  Ethereum財団とCantinaによる"Pectraセキュリティコンペ"で発見され、
  **メインネットへの影響が出る前に修正済み**。

---

## beacon-chain:slashing — スラッシング処理(2件)

- **[Lodestar] JavaScript側で`uint64`の値をnativeの`number`型として
  扱っていたことが原因で**、攻撃者が2^53を超える巨大な値を含む
  `AttesterSlashing`/`ProposerSlashing`をブロックに含めて送るだけで、
  浮動小数点の丸め誤差により一部クライアントだけがそれを不正だと
  誤判定し、**コンセンサス分裂**を引き起こせる脆弱性(CWE-190,
  fixed in v0.36.0)。

- もう1件(Lighthouse)はリリースノート内で開発者自身が「これらの
  バグはセキュリティリスクではなかった」と明言しており、実際には
  CLIフラグの回帰バグ(`--beacon-node`が無視される等)。severityの
  自動推定が誤っている例として記載。

## precompiles — プリコンパイルコントラクト(1件)

- **[Geth] プリコンパイル`0x04`(`dataCopy`)の実装がメモリ領域の
  浅いコピー(shallow copy)しか行っていなかったことが原因で**、攻撃者が
  「①メモリ領域Rに値Xを書く → ②`0x04`をRを引数に呼ぶ → ③Rを値Yで
  上書き → ④`RETURNDATACOPY`を呼ぶ」という順序の処理を行うコントラクト
  を呼び出すトランザクションを送るだけで、Geth系ノードだけが誤った
  値Yをスタックに積む結果となり、**コンセンサス分裂**(チェーン分岐)を
  引き起こせる脆弱性(`GHSA-69v6-xc2j-r2jf`, v1.9.17で修正)。

---

## txpool — トランザクションプール(1件)

- **[Geth] 将来実行予定(future)トランザクションプールの件数上限
  バリデーションが不十分だったことが原因で**、攻撃者が高ガス価格を
  付けた5,120件のfutureトランザクションを1メッセージにまとめて送る
  だけで、被害ノードの保留中トランザクションプール(mempool)を丸ごと
  追い出させDoSを引き起こせる脆弱性(geth ≤ 1.10.12)。

---

## database — ステート/DB層(1件、技術詳細は限定的)

- **[Geth] 2016年9月の"Shanghai DoS攻撃"** (`EXTCODESIZE`/`SLOAD`など、
  実コストに見合わない安価なガス代で状態肥大化・DB I/Oを圧迫させる
  一連の攻撃)を受けたホットフィックスリリース(v1.4.13)。データセット
  上のリリースノートは緩和策の告知のみで、具体的な差分は含まれていない
  — この一文は当時の公知の経緯を補足したもので、CSVの技術フィールド
  由来ではない点に注意。

## 除外・ノイズと判断したラベル/エントリ

以下は`severity=Critical/High`として記録されているが、(a) リリースノートの
定型文のみで攻撃手順を再構成できない、(b) 内容が明らかに脆弱性ではない
(スペック改名PR・機能追加・パフォーマンス改善など)、のいずれかに該当する
ため本文には含めなかった。存在の透明性のため一覧のみ記す。

| label | 該当件数(概算) | 除外理由 |
|---|---|---|
| `beacon-chain:execution-payload` | 3件中3件 | `prevRandao`へのリネーム、マージ関連テスト整備など、いずれも通常の仕様追随PRで脆弱性の記述なし |
| `beacon-chain:attestation` | 2件中2件 | validatorモニタのカーディナリティ改善、クロスクライアント相互運用のシリアライズ修正 — 攻撃シナリオの記述なし |
| `fork-choice` | 2件中2件 | Grandineの定型リリースノート、Lodestarの大型機能PR(Kintsugi実装)で個別バグの特定不可 |
| `beacon-chain:justification-and-finality` / `registry-updates` / `engine-api` / `build-ci` / `validator` | 各1件 | 機能PR、定型リリースノート、または起動オプション起因のローカルクラッシュ(リモート攻撃経路なし) |
| `other` の一部 | 7件中4〜5件 | 「重大なセキュリティ更新を行いました」等、内容の記述が無いリリースノート。Tekuの1件はCritical節のLog4Shellと同一リリース(21.12.1)の重複記録 |
| `p2p` の一部 | 9件中2件 | Bug Bounty経由で「詳細は後日公開」のまま技術詳細が非公開 |
| `sync` の一部 | 2件中1件 | EIP-8025の`MAX_PROOF_SIZE`定数調整 — 実測プルーフサイズに合わせた仕様パラメータのチューニングであり、脆弱性ではない |

## まとめ

- Critical/High 66件のうち、**再現可能なレベルまで攻撃手順を再構成できたのは
  概ね半数強**。残りはリリースノートの定型文や仕様追随PRで、severityの
  自動推定(LLM/ルールベース)には一定のノイズが混じっている
  ([`limitations.md`](./limitations.md) 参照)。
- 実際に**メインネットで悪用が確認されている**のは `GHSA-9856-9gg9-qcmq`
  (2021年8月、ブロック13107518でのマイノリティフォーク)のみ。
- 深刻な被害(コンセンサス分裂・資金喪失)に直結するバグは、EVMの
  メモリ管理(`RETURNDATA`/プリコンパイルの浅いコピー)、ガス計算の
  型変換、`selfdestruct`まわりの残高更新など、**「クライアント間で
  結果が食い違いうる箇所」に集中**している。
- 可用性(DoS)方向のバグは、devp2p/libp2pのリソース制限不備、
  RPC/GraphQLのクエリコスト無制限、txpoolの件数上限不備など、
  **「攻撃者が制御できる入力の量・サイズに対して上限が無い」箇所**に
  集中している。

*データソース: `data/ethereum_vulns.csv`。分類手法の詳細は
[`severity_labeling.md`](./severity_labeling.md)、既知の限界は
[`limitations.md`](./limitations.md) を参照。*
